# FlipCard — Style Guide & Design System

> **Purpose:** This document defines the complete visual and interaction design system for FlipCard, a flashcard web application built with Next.js and Tailwind CSS. It is intended to be consumed by AI coding assistants to ensure consistent, high-quality implementation across all components and pages.

---

## 1. Application Overview

### What It Is
FlipCard is a flashcard-style study and interview preparation tool. Users see a grid of cards, each displaying a word or phrase on its front face. Clicking a card triggers a flip animation that expands the card to fill the screen, revealing detailed content on the back — definitions, scenarios, talking points, or study notes.

### Target Users
- Job seekers preparing for behavioral interviews
- Students studying for exams
- Anyone who benefits from active recall using flashcards

### Core Interaction Pattern
1. User visits the **home page** (`/`) and sees a welcome screen with a "Create New Deck" button
2. User clicks **Create New Deck**, enters a name and optional description
3. App creates the deck and redirects to **`/deck/{unique-id}`** — a shareable URL
4. User sees an **empty card grid** with a prompt to add their first card
5. User **adds cards** (each card has a front face word/phrase and back face detail)
6. Cards can be organized by **category/tag** within the deck
7. User **clicks a card** → card **flips and expands** to full screen, revealing the back face
8. User reads the content, then **clicks to return** to the grid
9. The **deck URL** (`/deck/{unique-id}`) can be bookmarked or shared — anyone with the link can view and edit the deck
10. Returning to the **home page** shows a way to enter an existing deck URL or create a new one

### URL Structure
```
/                     → Landing page (create new deck or enter existing deck URL)
/deck/{deck-id}       → Card grid for a specific deck (main app experience)
```

The `deck-id` is a UUID generated by Supabase. Example: `/deck/a1b2c3d4-e5f6-7890-abcd-ef1234567890`

## 4. Typography

### Font Stack

| Role | Font | Fallback | Weight(s) | Load via |
|------|------|----------|-----------|----------|
| Headings & Card Fronts | **DM Sans** | system-ui, sans-serif | 500, 600, 700 | `next/font/google` |
| Body & Card Backs | **DM Sans** | system-ui, sans-serif | 400, 500 | `next/font/google` |
| Monospace (if needed) | **JetBrains Mono** | monospace | 400 | `next/font/google` |

> DM Sans is geometric, modern, and highly readable at all sizes — ideal for a minimal flashcard interface. Using a single family for both headings and body keeps things cohesive.

### Type Scale

| Token | Size | Line Height | Weight | Use |
|-------|------|-------------|--------|-----|
| `text-xs` | 12px / 0.75rem | 1.5 | 400 | Tags, metadata, timestamps |
| `text-sm` | 14px / 0.875rem | 1.5 | 400 | Secondary labels, helper text |
| `text-base` | 16px / 1rem | 1.625 | 400 | Body text, card back content |
| `text-lg` | 18px / 1.125rem | 1.625 | 500 | Card front text (short phrases) |
| `text-xl` | 20px / 1.25rem | 1.5 | 600 | Card front text (single words) |
| `text-2xl` | 24px / 1.5rem | 1.35 | 600 | Section headings |
| `text-3xl` | 30px / 1.875rem | 1.3 | 700 | Page title |
| `text-4xl` | 36px / 2.25rem | 1.2 | 700 | Expanded card title (detail view) |

### Typography Rules
- Card front text should be **centered** both horizontally and vertically
- Card front text uses `font-semibold` (600) or `font-bold` (700) depending on word length
- Card back body content uses `font-normal` (400), `text-base`, with `leading-relaxed` (line-height: 1.625)
- Maximum content width for card back text: `max-w-2xl` (672px) — centered on screen for readability
- Use `tracking-tight` on headings `text-2xl` and above
- Use `text-balance` on card front text to avoid awkward line breaks

### 6.3 Flip & Expand Animation

This is the signature interaction. Use **Framer Motion** (`motion` and `AnimatePresence`).

```
Animation Sequence (Card Click → Detail View):

1. FLIP (0ms → 300ms):
   - Card rotates on Y-axis: rotateY(0) → rotateY(90°)
   - Use perspective: 1000px on parent for 3D depth
   - Easing: ease-in (accelerate into the flip)

2. EXPAND (300ms → 600ms):
   - At rotateY(90°), swap content from front to back
   - Card expands from its grid position to full viewport
   - Use Framer Motion's layoutId for shared layout animation
   - Background fades to full-screen color
   - Easing: ease-out (decelerate into position)

3. CONTENT FADE-IN (500ms → 700ms):
   - Detail text fades in: opacity 0 → 1
   - Slight upward slide: translateY(12px) → translateY(0)
   - Easing: ease-out

Animation Sequence (Close → Back to Grid):
   - Reverse of above
   - Content fades out first (0ms → 150ms)
   - Card shrinks back to grid position (150ms → 450ms)
   - Card flips back to front face (450ms → 600ms)

Framer Motion Implementation Notes:
  - Use <AnimatePresence> to handle mount/unmount
  - Use layoutId={card.id} on both the grid card and the detail view 
    for automatic shared layout transitions
  - Apply perspective via style={{ perspective: 1000 }} on grid container
  - Use transform-style: preserve-3d on the card element
  - Hardware accelerate with will-change: transform

Alternative (Simpler) Approach:
  If full 3D flip is too complex, use this sequence instead:
  1. Card scales up slightly and fades (200ms)
  2. Full-screen overlay slides up from card position (300ms)
  3. Content fades in (200ms)
  This is easier to implement and still feels polished.
```

## 9. Motion & Transitions

### General Principles
- Every interactive element should have a transition (even if subtle)
- Default transition: `transition-all duration-200 ease-out`
- Hover scale effects: use `scale-[1.02]` — subtle, not bouncy
- Avoid animations longer than 600ms total
- Respect `prefers-reduced-motion`: disable flip animation, use simple fade instead

### Specific Transitions

| Element | Property | Duration | Easing |
|---------|----------|----------|--------|
| Card hover | transform, shadow, border | 200ms | ease-out |
| Card press | transform | 100ms | ease-in |
| Card flip | rotateY | 300ms | ease-in-out |
| Card expand | layout position + size | 300ms | spring (Framer) |
| Detail content appear | opacity, translateY | 200ms | ease-out |
| Tag filter select | background-color, color | 150ms | ease-out |
| Dark mode toggle | all colors | 200ms | ease-out |
| Page load card stagger | opacity, translateY | 300ms each, 50ms stagger | ease-out |

### Page Load Animation
Cards should stagger in on initial page load:
```
Each card:
  - Initial: opacity: 0, translateY: 12px
  - Animate: opacity: 1, translateY: 0
  - Duration: 300ms
  - Stagger: 50ms delay between each card
  - Max stagger: cap at ~400ms total (don't delay cards too long)
```

## 10. Accessibility

### Requirements
- All interactive elements must be keyboard accessible
- Cards must be `<button>` elements (not `<div onClick>`)
- Focus states must be visible: `ring-2 ring-primary-500 ring-offset-2`
- Color contrast: minimum 4.5:1 for body text, 3:1 for large text
- Expanded card view must trap focus (focus trap)
- Escape key closes expanded card view
- Category filter tags should use `role="tablist"` / `role="tab"` pattern
- Include `aria-expanded` on cards to indicate open/closed state
- Include `aria-label` on icon-only buttons (close, dark mode toggle)
- Respect `prefers-reduced-motion` — replace flip with fade
- Minimum touch target: 44x44px on all interactive elements

### Screen Reader Announcements
- When card expands: announce "Viewing details for {card title}"
- When card closes: announce "Returned to card grid"
- Use `aria-live="polite"` region for filter changes: "Showing {n} cards in {category}"

## 12. Dark Mode Implementation

### Strategy
- Use Tailwind's `class` strategy (not `media`)
- Store preference in `localStorage` under key `"theme"`
- On load: check localStorage → fallback to `prefers-color-scheme`
- Toggle adds/removes `dark` class on `<html>` element
- Prevent flash of wrong theme (FOUC): inline script in `<head>` that sets class before render

### Tailwind Config
```js
// tailwind.config.js
module.exports = {
  darkMode: 'class',
  // ... rest of config
}
```

### Database Schema (SQL)

Run this in the Supabase SQL Editor to create the tables:

```sql
-- Decks table (top-level grouping, identified by UUID in the URL)
CREATE TABLE decks (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- Categories table (scoped to a deck)
CREATE TABLE categories (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  deck_id UUID NOT NULL REFERENCES decks(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  color TEXT DEFAULT 'blue',  -- maps to predefined tag color names
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(deck_id, name)       -- category names unique within a deck
);

-- Flashcards table (belongs to a deck, optionally tagged with a category)
CREATE TABLE flashcards (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  deck_id UUID NOT NULL REFERENCES decks(id) ON DELETE CASCADE,
  title TEXT NOT NULL,                    -- front face text (word or short phrase)
  content TEXT NOT NULL,                  -- back face text (detailed content, supports markdown)
  category_id UUID REFERENCES categories(id) ON DELETE SET NULL,
  sort_order INTEGER DEFAULT 0,          -- for manual reordering later
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- Indexes
CREATE INDEX idx_flashcards_deck ON flashcards(deck_id);
CREATE INDEX idx_flashcards_category ON flashcards(category_id);
CREATE INDEX idx_categories_deck ON categories(deck_id);

-- Auto-update updated_at timestamp
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER flashcards_updated_at
  BEFORE UPDATE ON flashcards
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at();

CREATE TRIGGER decks_updated_at
  BEFORE UPDATE ON decks
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at();

-- Row Level Security (RLS) — permissive for v1 (no auth)
-- When auth is added later, add user_id to decks table and update policies
ALTER TABLE decks ENABLE ROW LEVEL SECURITY;
ALTER TABLE flashcards ENABLE ROW LEVEL SECURITY;
ALTER TABLE categories ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Allow all access to decks" ON decks
  FOR ALL USING (true) WITH CHECK (true);

CREATE POLICY "Allow all access to flashcards" ON flashcards
  FOR ALL USING (true) WITH CHECK (true);

CREATE POLICY "Allow all access to categories" ON categories
  FOR ALL USING (true) WITH CHECK (true);
```

**Auth-ready note:** When authentication is added later, add a `user_id UUID REFERENCES auth.users(id)` column to the `decks` table and update the RLS policy to `USING (auth.uid() = user_id)`. Cards and categories inherit access through their `deck_id` foreign key — you can use a subquery policy like `USING (deck_id IN (SELECT id FROM decks WHERE user_id = auth.uid()))`. The schema is designed so this is a non-breaking migration.

### Future: Authentication & Permissions Plan

This section is NOT implemented in v1. It documents the intended future architecture so that v1 decisions don't create obstacles. **AI assistants building v1 should NOT implement any of this — just be aware of it.**

**Phase 1 — Basic Auth (owner-only editing):**
1. Enable Supabase Auth (email/password, or OAuth providers like Google/GitHub)
2. Add `user_id UUID REFERENCES auth.users(id)` to `decks` table
3. Set `user_id` automatically when a deck is created (from `auth.uid()`)
4. Update RLS policies:
   - Decks: anyone can SELECT (read/view), only owner can INSERT/UPDATE/DELETE
   - Cards & Categories: inherit from deck ownership via subquery
5. UI changes: show edit/delete controls only to the deck owner, read-only view for everyone else

**Phase 2 — Shared permissions (invite collaborators):**
1. Add a `deck_permissions` table:
   ```sql
   -- FUTURE: Do not create this table in v1
   CREATE TABLE deck_permissions (
     id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
     deck_id UUID NOT NULL REFERENCES decks(id) ON DELETE CASCADE,
     user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
     role TEXT NOT NULL DEFAULT 'viewer',  -- 'viewer' | 'editor'
     created_at TIMESTAMPTZ DEFAULT now(),
     UNIQUE(deck_id, user_id)
   );
   ```
2. Update RLS policies to check both ownership AND the permissions table
3. UI changes: add a "Share" button on the deck page, invite by email, manage permissions

**v1 architectural safeguards (DO implement these):**
- All card/category mutations go through server actions (not direct client-side Supabase calls) — this makes it easy to add auth checks in one place later
- The deck ID in the URL is a UUID (not a sequential integer) — this provides basic security through obscurity for v1, since deck URLs are unguessable
- RLS is enabled on all tables even though policies are permissive — flipping to restrictive later is a policy change, not a schema change
- Keep the Supabase anon key as `NEXT_PUBLIC_` for reads, but plan to use a `SUPABASE_SERVICE_ROLE_KEY` (server-only) for writes when auth is added

### Local Fallback Mode (No Database Required)

**IMPORTANT:** The app MUST be able to run with `npm run dev` without Supabase configured. This allows the UI, animations, and components to be developed and tested before setting up a database.

**How it works:**
- Check `isSupabaseConfigured` at the data layer
- If `false`, use an in-memory store (a plain JS Map/object) that persists for the duration of the dev server session
- All CRUD operations work identically — same types, same function signatures — they just read/write to memory instead of Supabase
- The in-memory store should be pre-seeded with sample data so the UI isn't empty on first run

**Implementation pattern:**


## 14. CRUD Operations & API Layer

### Data Fetching Pattern

Use Next.js App Router conventions:
- **Read** operations: Server Components fetch data directly (no API route needed)
- **Write** operations: Server Actions or Route Handlers for create/update/delete
- **Revalidation**: Use `revalidatePath('/deck/[deckId]')` after mutations to refresh the card grid
- **All card/category operations are scoped to a deck** — always pass `deckId`

### Server Actions

### Query Patterns

```typescript
// Fetch a single deck by ID (for the /deck/[deckId] page)
const { data: deck } = await supabase
  .from('decks')
  .select('*')
  .eq('id', deckId)
  .single();

// Fetch all cards in a deck with their category (joined)
const { data: cards } = await supabase
  .from('flashcards')
  .select('*, category:categories(*)')
  .eq('deck_id', deckId)
  .order('sort_order', { ascending: true });

// Fetch cards in a deck filtered by category
const { data: cards } = await supabase
  .from('flashcards')
  .select('*, category:categories(*)')
  .eq('deck_id', deckId)
  .eq('category_id', categoryId)
  .order('sort_order', { ascending: true });

// Fetch all categories in a deck
const { data: categories } = await supabase
  .from('categories')
  .select('*')
  .eq('deck_id', deckId)
  .order('name', { ascending: true });
```

## 18. Deployment (Vercel)

### Setup
- Connect GitHub repo to Vercel
- Framework preset: Next.js (auto-detected)

- Output directory: `.next` (default)

### Environment Variables (set in Vercel Dashboard → Settings → Environment Variables)
```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
```

### Notes
- `NEXT_PUBLIC_` prefix makes these available client-side (required for Supabase JS client)
- For production, consider adding a `SUPABASE_SERVICE_ROLE_KEY` (server-only, no `NEXT_PUBLIC_` prefix) for server actions that need elevated permissions
- Vercel automatically provides HTTPS, CDN, and edge caching
- Supabase and Vercel are in the same ecosystem — low latency when both use AWS us-east-1

## 20. Key Design Principles (Summary for AI Assistants)

1. **Content is king.** The UI should disappear — cards and their content are the focus. Minimal chrome.
2. **The flip is the moment.** The card flip/expand animation is the signature interaction. It should feel satisfying and smooth.
3. **Quiet confidence.** Blue and slate palette, clean type, generous whitespace. No gradients, no noise, no decorative elements.
4. **Respect the user.** Accessible, keyboard-friendly, responsive, fast. Respect motion preferences.
5. **Dark mode is first-class.** Not an afterthought — test every component in both modes.
6. **Consistent spacing.** Use the Tailwind scale. When in doubt, use `gap-6` for grid items, `p-6` for card padding, `space-y-4` for content flow.
7. **One font family.** DM Sans for everything. Differentiate with weight and size, not font changes.
8. **CRUD should feel invisible.** Adding, editing, and deleting cards should be fast and frictionless. Modals for create/edit, inline menus for actions, toast confirmations — never redirect to a separate page.
9. **Optimistic updates.** When a user saves or deletes a card, update the UI immediately and sync with Supabase in the background. Roll back on error.
10. **Auth-ready architecture.** No authentication in v1, but the database schema includes RLS policies, mutations go through server actions (single place to add auth checks later), and the future permissions model (owner/editor/viewer) is documented. UUIDs in URLs provide security through obscurity until auth is added.
